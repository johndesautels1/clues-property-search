  2. PublicRemarks - How to Display Without Breaking 181 Schema?

  Current State:
  - Bridge returns PublicRemarks (long description text)
  - Currently used for TEXT PARSING to extract:
    - Field 48 (interior condition)
    - Field 59 (renovations)
    - Field 134 (smart home features)
    - Field 135 (accessibility)
    - Field 138 (special assessments)
  - After extraction, "cleaned remarks" are generated (extracted sentences removed)

  Options:

  Option A: Store as Non-Numbered Metadata Field

  // In Property interface (not in 181 fields)
  property.metadata = {
    publicRemarks: string,          // Full original remarks
    publicRemarksExtracted: string, // Cleaned version (extracted data removed)
    virtualTourURL: string,         // Already doing this
    extendedMLSData: object,        // Already doing this
  }
  Pros:
  - Doesn't break 181-field schema
  - Already have precedent (extendedMLSData, virtualTourURL)
  - Can display in UI without numbering
  - Can still use for text mining

  Cons:
  - Not part of "canonical" fields
  - Harder to search/filter by remarks content

  Option B: Add as Field 182 (Schema Extension)

  - Make PublicRemarks field #182
  - Accept that schema is now 182 fields
  - Update all documentation

  Pros:
  - Treats remarks as first-class data
  - Searchable like other fields

  Cons:
  - Breaks "181 fields" branding
  - Slippery slope - what about photos, virtual tours, etc.?

  Option C: Keep Current Approach (Parsing Only)

  - Continue using PublicRemarks ONLY for extraction
  - Display "cleaned remarks" in PropertyCard
  - Don't store as separate field

  Pros:
  - No schema changes
  - Already working

  Cons:
  - Loses full original remarks if cleaning is imperfect
  - Can't search original MLS description

  Recommendation: Option A - Store as property.metadata.publicRemarks alongside extendedMLSData
